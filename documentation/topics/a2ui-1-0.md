# A2UI 1.0

AshA2ui speaks two versions of the A2UI protocol. Every surface defaults to
v0.9.1 (nothing changes for existing consumers), and opting a surface into
**v1.0** is one DSL line:

```elixir
a2ui do
  spec_version "1.0"

  component :table do
    fields [:name, :status]
  end
end
```

Dynamic (agent-composed) surfaces opt in per resolve:

```elixir
AshA2ui.Dynamic.resolve(spec, allowlist: allowlist, spec_version: "1.0")
```

Everything else — components, fields, queries, contexts, nested forms,
actions, authorization — is identical between the versions. What changes is
the **wire contract**, and each change exists to fix a real UX failing of
the 0.9 protocol. This topic explains the contract and the rationale;
the conformance evidence lives in the test suite (see
"The executable spec" below).

> #### v1.0 is a Release Candidate {: .warning}
>
> A2UI v1.0 is published as an RC. AshA2ui vendors the v1.0 schemas, the
> basic catalog, and the upstream conformance cases from google/A2UI at a
> pinned commit (`priv/a2ui/v1_0/NOTES.md` records it), so drift between
> the RC and the final spec is a re-vendor + test run away from being
> visible.

## What v1.0 actually changes (verified against the spec)

| Change | Spec mechanics |
|---|---|
| Inline `createSurface` | `createSurface` may carry `components` and `dataModel` — a whole surface in one message |
| `actionResponse` | Client actions may set `wantResponse: true` + a unique `actionId`; the server answers with an `actionResponse` message (`{"value": ...}` or `{"error": {code, message}}`) echoing the id |
| `callFunction` / `functionResponse` | Server→client RPC invoking catalog functions, correlated by `functionCallId` |
| `surfaceProperties` | Replaces the v0.9 `theme` block; `primaryColor` is gone — visual styling is deferred entirely to the renderer |
| Catalog `functions` map | Function definitions are an object keyed by name (was a list); standard JSON Schema metadata fields allowed |
| UAX #31 identifiers | Catalog entity names (components, functions, argument/property names) must match `^[\p{XID_Start}_][\p{XID_Continue}]*$` |
| `@index` builtin | List templates may call `@index` for the item's 0-based position; the `@` prefix is reserved for system functions |
| Deletion semantics | `updateDataModel` **without** a `value` key deletes the key at `path` (an omitted value is destructive — every AshA2ui message always carries `value`) |
| Text variants | The basic catalog's `Text` keeps only `caption`/`body`; the `h1`–`h5` variants are gone — headings are Markdown |
| Wire function calls | `returnType`/`callableFrom` are catalog metadata only, no longer allowed on wire-level FunctionCall payloads |

## The UX rationale, feature by feature

### `actionResponse` — feedback tied to the action, not the surface

The v0.9.1 protocol is fire-and-forget: the client sends an `action`, and
*eventually* data-model writes arrive. Nothing correlates a click with its
outcome, so the emitted surfaces route all feedback through global reserved
paths (`/ui/status`, `/ui/action_result`) — a flash bar. Two clicks in
flight are indistinguishable, and "is it still working?" is unanswerable.

On a v1.0 surface:

- every emitted action event carries `wantResponse: true`;
- the client attaches a unique `actionId`;
- `AshA2ui.ActionHandler` answers **first** with an `actionResponse`
  echoing that id — `{"value": <the /ui/response object>}` on success, or
  `{"error": {"code": ..., "message": ...}}` with a stable machine-readable
  code (`VALIDATION_FAILED`, `UNAUTHORIZED`, `INVALID_ACTION`,
  `ACTION_FAILED`) on failure — followed by the usual data-model
  follow-ups.

The shipped JS hook completes the loop with **0-RTT pending feedback**: the
moment a v1.0 action fires it writes `{"status": "pending", "message":
"Working…"}` into the surface's `/ui/response` locally, so the bound status
component reacts before the network does; the server's response then
overwrites it, and a watchdog turns a never-answered action into a visible
timeout error instead of silence. The hook also re-dispatches every
response as a bubbling `"ash-a2ui:action-response"` DOM event for
host-level integration (toasts, analytics, optimistic UI).

### `/ui/response` — one structured status object

With per-action responses in place, the v0.9.1 trio of reserved paths
(`/ui/status` text, `/ui/action_result` map, `/ui/action_result_text`)
collapses into a single structured object:

```json
{
  "status": "ok" | "error" | "pending" | "",
  "code": "VALIDATION_FAILED",
  "message": "Created successfully.",
  "result": { "secret": "s3cr3t" },
  "resultText": "Secret: s3cr3t"
}
```

written at `/ui/response` after every handled action (the `code` key is
present on errors). The emitted components keep their ids (`status_text`,
`action_result_text`) and simply bind the subpaths — custom catalogs and
themes keep working. Programmatic consumers stop scraping display text: the
`actionResponse` carries the same object, so "did *my* action succeed" is a
field access. All other reserved paths
([Data Model Conventions](data-model-conventions.md)) are unchanged.

### Inline `createSurface` — one-message surfaces

v0.9.1 bootstraps a surface as three messages (`createSurface` →
`updateComponents` → `updateDataModel`). Renderers are allowed to paint
between them — an empty-shell flash — and every transport has to carry a
three-part handshake (awkward for HTTP responses and agent-tool outputs,
where "the UI" wants to be *one value*).

A v1.0 surface bootstraps as a **single** `createSurface` message carrying
the full component list and the initial data model. One paint, one payload.
This is exactly what agent-composed panels want: an LLM tool result or an
HTTP response body is one self-contained message
(`AshA2ui.Dynamic.build_surface/2` returns a one-element list for v1.0
surfaces).

### `callFunction` — a minimal, honest server→client escape hatch

v1.0 lets the server invoke catalog functions on the client. AshA2ui
deliberately keeps this surface **small**: nothing in the framework emits
`callFunction` automatically. `AshA2ui.Encoder.V1_0.call_function/3` builds
a spec-valid message when *you* decide an imperative client effect is
warranted (e.g. `openUrl`), and the shipped hook executes it against a
host-registered function table (`configureAshA2ui({functions: {...}})`,
`openUrl` built in) and returns the `functionResponse` when asked.

The restraint is the point: function calls are imperative and leave no
trace in the data model. Anything expressible as data — status, errors,
selection state — should stay `updateDataModel`, which every renderer
replays correctly after a reconnect. Candidates like focus management and
scroll restoration remain documented-but-deferred until the upstream
renderer exposes the corresponding client functions.

### `surfaceProperties` — branding metadata without styling opinions

v1.0 renames the v0.9 `theme` block to `surfaceProperties` and drops
`primaryColor`: the protocol stops pretending to own visual style. AshA2ui
never used `theme.primaryColor` — theming is entirely CSS variables on the
renderer side (see [Theming](theming.md)) — so this lands as pure gain:
pass `surface_properties: %{"agentDisplayName" => "Support Agent"}` to
`AshA2ui.Info.build_surface/2` and it travels in the `createSurface`
payload, validated against the catalog's `surfaceProperties` schema.

### Markdown headings

The v1.0 basic catalog removed the `h1`–`h5` `Text` variants (headings are
Markdown now — render `Text` through a Markdown parser, per the basic
catalog implementation guide). The v1.0 encoder converts automatically:
literal heading text becomes `"## Heading"`, and data-bound headings (e.g.
card titles) interpolate through the catalog's `formatString` function.
Configure the hook's `markdown` option so headings render as headings —
already recommended for v0.9.1 surfaces.

### Deletion semantics, `@index`, UAX #31 — the audit

- **Deletion**: in v1.0, an `updateDataModel` *omitting* `value` deletes
  the key at `path`. AshA2ui never relied on implicit deletion — every
  emitted message carries an explicit `value` (asserted by the conformance
  suite), and region resets are explicit whole-value writes (`/form` to its
  initial shape, `/errors` to `{}`).
- **UAX #31**: every generated component id, action name, and function
  name is ASCII snake_case — a strict subset of the required
  `XID_Start`/`XID_Continue` classes. The conformance suite asserts this
  over the full kitchen-sink component tree.
- **`@index`**: available to custom catalogs; the emitted row templates
  identify records by id (not position), so nothing needed it. Adopted
  nowhere, deliberately — positional identity is fragile under refresh.

## The executable spec

"1.0-conformant" is a claim you should be able to *run*. The test suite
contains three layers:

1. **Vendored upstream conformance cases** (`test/v1_0_conformance_test.exs`)
   — the spec's own schema test suites run against the vendored v1.0
   schemas through `AshA2ui.Test.SchemaHelper`. Twelve upstream-invalid
   cases are only detectable via draft-2020-12's `unevaluatedProperties`,
   which the draft-7 validator ignores; each is pinned in
   `@draft7_invisible_failures` with the suite failing if the blind spot
   ever heals silently.
2. **Encoder conformance** (`test/v1_0_encoder_test.exs`) — every message
   `AshA2ui.Encoder.V1_0` emits validates against the v1.0 schemas, plus
   the semantic rules schemas can't express: single inline bootstrap,
   `wantResponse` on every action, no wire `returnType`/`callableFrom`,
   explicit `value` on every `updateDataModel`, identifier rules on all
   generated names.
3. **Handler conformance** (`test/v1_0_action_handler_test.exs`) — the
   `actionResponse` contract round-trip: id echo, value/error shapes,
   stable error codes, `/ui/response` mirroring, and v0.9.1 isolation
   (byte-identical legacy output).

## Renderer status (read before flipping production surfaces)

As of `@a2ui/lit` 0.10.1 / `@a2ui/web_core` 0.10.4, **no published npm
release ships a v1.0 runtime** — the packages expose `v0_8`/`v0_9` entry
points only (`web_core` carries v1.0 *schemas*, no processor). The shipped
`priv/js/ash_a2ui_hook.js` is therefore the v1.0-capable client layer: it
detects v1.0 messages, expands the inline `createSurface` into the triple
the v0_9 renderer understands, generates the `actionId`/`wantResponse`
handshake with optimistic pending state, and executes `callFunction`
against the host function table. When upstream ships a v1_0 renderer entry
point, the hook's adapter seam is the only thing that changes.

## Choosing a version

- **New surfaces**: use `spec_version "1.0"` with the shipped hook — you
  get per-action feedback and one-message bootstraps at no cost.
- **Existing surfaces**: v0.9.1 output is frozen and byte-identical;
  upgrade surface-by-surface whenever you like. The version is per-surface,
  so mixed deployments are fine.
- **Custom renderers/transports**: v1.0 halves your integration surface
  (one bootstrap message, one status object, correlated responses), but
  make sure your renderer accepts inline `createSurface` first.
